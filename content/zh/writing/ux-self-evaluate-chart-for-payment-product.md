| 主题 | 关键点 |
| --- | --- |
| **新用户指引** | 新用户可以理解页面文案 |
|  | 新用户可以看到全部的可用渠道 |
|  | 新用户可以快速选择自己能用的渠道 |
| **老用户指引** | 老用户可以默认选择之前使用的渠道 |
|  | 老用户可以选择其它的渠道 |
| **渠道查询** | 用户可以在支付前提前准备好可用渠道 |
|  | 用户可以对比不同渠道 |
|  | 系统可以推荐优质渠道 |
|  | 单一渠道可用时，用户可以跳过渠道选择 |
|  | 渠道不可用时，用户可以知道渠道不可用原因 |
| **银行卡渠道** | 用户可以选择不储存银行卡 |
|  | 卡号的输入可以使用OCR |
|  | 卡号输入的同时，可以快速识别卡BIN |
|  | 卡渠道需要输入的字段，是符合合规要求的最小集 |
| **安全感** | 产品接入核身风控产品（支付密码、指纹识别、OTP验证等） |
|  | 智能识别，高科技感的输入（银行卡、身份证件的OCR识别等） |
|  | 支付信息截屏屏蔽，码动态更新 |
|  | 免密协议的签署需要明确，用户要有感知 |
|  | 支付并绑卡的确认 |
|  | 数字键盘是否有安全标记 |
|  | 支付流程中是否有其余的安全认证 |
| **交互细节** | 部分场景中数字键盘免点击，减少用户的点击操作 |
|  | 数字键盘添加常用的金额，减少用户的点击操作 |
| **渠道展示** | 渠道的名称 |
|  | 渠道的受理标识 |
|  | 渠道的运营信息展示 |
|  | 渠道的可用状态 |
|  | 渠道的使用说明（操作流程简介） |
|  | 渠道的使用预期 |
| **交易信息展示** | 需要支付的交易信息，需要很明确 |
|  | 金额信息需要很突出 |
|  | 运营信息需要很明确 |
|  | 收款方信息需要很明确 |
|  | 出资方信息建议可以显示 |
|  | 交易有效期的时间需要很突出 |
|  | 汇率信息需要很明确 |
| **营销信息展示** | 营销信息连贯展示，没有断点 |
|  | 营销信息符合当地文化，没有歧义 |
|  | 营销信息展示清晰明确 |
| **支付结果** | 支付失败后可以重新二次支付 |
|  | 支付成功后可以有新的转化 |
| **数据分析** | 消费场景下的支付漏斗数据 |
|  | 支付意愿率分析 |
|  | 新老用户拆分分析 |
|  | 渠道成功率分析 |
|  | 渠道用户占比分析 |
|  | 支付失败原因分析 |
